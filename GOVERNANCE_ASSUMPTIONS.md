# Governance Assumptions

These assumptions define the current public validation boundary for ZT&SI Stability Gateway.

## Runtime Assumptions

- Every candidate output is provisional until evaluated.
- Final manifestation must pass drift, coherence, policy, governance, firewall, lineage, memory, and telemetry handling.
- Stabilization is bounded and public-demonstrator level only.
- Blocked outputs must not win arbitration.
- Runtime memory and logs are local observability artifacts and must remain untracked.

## Policy Assumptions

- Policy thresholds are configurable but must remain explicit.
- Contradictory policy settings are possible and must be stress-tested.
- Severity escalation should be observable.
- Lockdown behavior is a governance state, not a claim of complete production security.

## Replay Assumptions

- Historical events may be incomplete, malformed, or generated by older runtime versions.
- Replay compares outcomes; it does not prove semantic correctness.
- Replay mismatches are validation findings, not automatic defects.
- Lineage reconstruction depends on persisted memory and graph records.

## Telemetry Assumptions

- Telemetry is used for observability and validation, not as the sole source of truth.
- Missing event fields degrade confidence.
- Orphan lineage ids indicate a runtime history integrity concern.
- Runtime state desync should block release readiness until understood.

## Agent Mesh Assumptions

- Agents are sandboxed cognition processes.
- Agent permissions, drift budgets, recursion quotas, memory scope, lineage scope, and output rights are explicit constraints.
- Agent conflict is resolved by existing public arbitration behavior only.
- Proprietary arbitration methods are outside this public repository.

## Boundary Assumptions

- This repository is public.
- Private sovereign-core mechanics do not belong here.
- Secrets, credentials, runtime dumps, and local machine paths do not belong here.
- Public validation should prove operational discipline without exposing protected research.
